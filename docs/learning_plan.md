# anti_drone_agc_workspace 后续研发与学习规划

## 3. 后续该做什么

下面这条路线是按优先级排的，不是按技术炫酷程度排的。

### 第一阶段：把现有功能包打磨成稳定子系统

**目标**

- 让 `drone_launch_controller`、`lqr_trajectory_tracker`、`air_ground_core` 各自边界清晰、接口一致、可稳定复现

**具体要做什么**

1. 收敛 `drone_launch_controller` 的接口定义
2. 检查 `README`、`launch`、`srv/msg`、源码实现是否一致
3. 收敛 `lqr_trajectory_tracker` 的职责，避免把任务管理、飞控模式切换、控制器逻辑过度耦合在一个节点里
4. 给地面车控制补最小可配置能力，而不是固定圆周运动
5. 补测试脚本、参数说明、运行说明

**这一阶段你要补的知识**

- ROS 节点职责划分
- 接口设计与文档一致性
- 控制系统工程化，不只是算法本身
- 参数化 launch 和实验复现方法

**阶段产出文件**

- `docs/subsystem_refactor_plan.md`
- `docs/drone_controller_api.md`
- `docs/lqr_tracker_api.md`
- `docs/ground_vehicle_control_api.md`
- `scripts/run_takeoff_demo.sh`
- `scripts/run_lqr_demo.sh`

**完成标准**

- 每个核心包的输入输出可以一页纸说清楚
- 文档描述和实际接口一致
- 各模块可以独立启动和验证

---

### 第二阶段：补一个任务调度层，形成真正的“空地协同主流程”

**目标**

- 不再是“起飞模块 + 跟踪模块 + 车模块”，而是有一个明确任务主线

**具体要做什么**

1. 设计一个主控节点或任务管理节点
2. 定义任务状态机，例如：
   - 系统初始化
   - 仿真环境就绪
   - 地面车启动
   - 无人机连接确认
   - 无人机起飞
   - 轨迹跟踪
   - 目标搜索/巡逻
   - 返回或降落
3. 明确谁发命令，谁回状态，失败时怎么处理
4. 设计最小闭环任务场景，不要一开始做大而全

**建议的最小任务场景**

- 地面车按给定路径移动
- 无人机起飞到指定高度
- 无人机执行固定轨迹或伴随跟踪
- 系统持续发布统一状态
- 任务完成后悬停或降落

**这一阶段你要补的知识**

- 状态机设计
- 多节点协同
- ROS 系统架构设计
- 异常流程处理

**阶段产出文件**

- `docs/system_architecture_v1.md`
- `docs/mission_state_machine.md`
- `docs/coordination_interfaces.md`
- `src/air_ground_core/src/mission_manager.cpp` 或等价任务节点
- `launch/system_demo.launch`

**完成标准**

- 系统存在一个统一的“主流程”
- 模块之间的协同关系是显式的，不靠人工手动拼
- 可以演示一次最小空地协同任务

### 第三阶段：把 LQR 从“能飞”推进到“能比较、能分析、能写结果”

**目标**

- 让轨迹跟踪部分具备实验价值和报告价值，而不是只停在演示

**具体要做什么**

1. 固化实验场景
2. 固化对比指标
3. 系统性调参，而不是凭感觉调
4. 输出结果文件，而不是只看 RViz

**建议实验维度**

- 轨迹类型对比：`linear`、`circular`、`fixed_horizontal`、`random`
- 参数对比：不同 `Q/R`
- 初始扰动对比：初始位置偏差、速度偏差
- 噪声影响对比
- 最大误差、平均误差、稳态误差、收敛时间

**这一阶段你要补的知识**

- LQR 参数整定方法
- 实验设计
- 误差指标定义
- 数据记录与可视化

**阶段产出文件**

- `docs/lqr_experiment_design.md`
- `docs/lqr_result_analysis.md`
- `results/lqr/`
- `scripts/run_lqr_experiments.sh`
- `scripts/analyze_trajectory.py` 的增强版结果输出

**完成标准**

- 同一个实验能稳定复现
- 至少能拿出一组有说服力的参数对比结论
- 有图、有表、有日志，而不是只有口头结论

---

### 第四阶段：把感知模块真正接入主链路

**目标**

- 让 A-LOAM 和 Velodyne 仿真从“仓库里的组件”变成“系统使用的组件”

**具体要做什么**

1. 明确感知模块输入输出
2. 选择它在当前项目中的最小用途
3. 决定它是服务于定位、环境感知，还是目标搜索
4. 把感知结果接到任务层，而不是只单独显示在 RViz

**建议你先做的最小版本**

- 先不做真正复杂的反无人机识别
- 先把 SLAM/里程计作为位姿来源或环境建图能力使用
- 让任务层能消费一类感知结果，例如：
  - 当前位姿质量
  - 已建图区域
  - 障碍物位置

**这一阶段你要补的知识**

- TF 树
- 点云处理基础
- LOAM 输入输出链路
- 感知结果如何进入任务决策

**阶段产出文件**

- `docs/perception_pipeline.md`
- `docs/tf_tree_design.md`
- `docs/slam_integration_plan.md`
- `launch/perception_demo.launch`

**完成标准**

- 感知模块在主流程中有明确作用
- 不是单独运行一个 A-LOAM demo，而是接入整体系统

## 4. 对应要学什么

不是从头系统学 ROS，而是按后续研发需要补短板。

### 4.1 你最应该优先补的

1. ROS 系统架构设计  
不是再学怎么写 `publisher/subscriber`，而是学怎么划分节点职责、接口边界、启动关系。

2. 状态机和任务调度  
因为你现在最缺的是“统一任务主线”。

3. 控制实验设计  
因为 LQR 继续往前走，已经不是“控制公式会不会写”的问题，而是“实验结果怎么证明你的设计有效”。

4. 工程化文档与复现  
你这个项目已经到了必须重视“别人能不能跑、以后你自己能不能复现”的阶段。

### 4.2 第二优先级补的

1. TF 与坐标系管理
2. 点云与 LOAM 基础
3. PX4/MAVROS 进阶模式切换和故障保护
4. ROS 参数系统和 launch 结构整理

### 4.3 暂时不该优先补的

1. 过深的 SLAM 数学细节
2. 过早引入复杂视觉识别
3. 一上来重写所有包
4. 一上来追求多无人机、多车、多目标

## 6. 一个更实际的判断

如果你的目标是“把这个项目继续做成一个像样的系统”，那你后面最该做的不是再单独写一个新算法包，而是优先做下面三件事：

1. 做任务调度层
2. 做实验与结果闭环
3. 做感知接入的最小主流程

如果你的目标是“做出一篇像样的报告或比赛展示”，那优先级则是：

1. 稳定演示链路
2. 固化 LQR 实验结果
3. 用状态机把空地协同讲清楚
4. 最后再决定感知模块接到哪里最有说服力

## 7. 最后的建议

你现在不是缺“继续学什么框架”，你是缺“给这个项目定下一阶段目标”。  
从仓库现状看，最合理的下一阶段目标应该是：

**把现有控制能力整合成一个最小空地协同任务系统，并且能稳定演示、稳定复现、稳定产出实验结果。**

如果只给一句最直接的路线：

1. 先做任务调度层
2. 再做 LQR 实验闭环
3. 再把感知接到主链路
4. 最后做统一 demo 和报告材料

这条路线比继续零散加功能更稳，也更容易形成真正有价值的成果。
